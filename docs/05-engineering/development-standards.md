# 开发规范

## 基本原则

- 新增业务能力先定义业务流程和 API 契约，再实现端侧页面。
- 共享类型优先放入 `packages/irrigation-api`。
- 领域纯函数优先放入 `packages/irrigation-domain`。
- demo 能力必须与 production 能力显式隔离。
- 关键写操作必须有审计或事件记录规划。

## API-first 规范

- 小程序所有业务请求走服务端 API。
- Web 新增写操作优先走服务端 API，除非明确说明保留 Supabase 直连。
- API 响应统一使用 `{ data }`，错误统一使用 `{ error, code? }`。
- 修改 API 必须同步更新端侧调用、共享类型和文档。

## demo/production 隔离

- demo 设备必须标记 `source=demo`。
- 正式列表默认不混入 demo 数据。
- demo seed 不允许在生产环境自动执行。
- demo 配置不得成为正式数据模型的必填字段。

## 文档同步要求

文档同步是每个功能迭代的强制验收项。只要功能、接口、数据模型、部署方式、排障方式或路线图状态发生变化，相关文档必须与代码同批更新；不能把文档更新留到“以后补”。

每个功能迭代必须检查：

- 是否更新模块文档。
- 是否更新缺口分析状态。
- 是否更新迭代追踪。
- 是否影响 API 契约或数据模型。
- 是否需要新增排障说明。

合并前必须满足：

- 已完成的缺口在 `docs/04-roadmap/gap-analysis.md` 中更新状态。
- 本次迭代在 `docs/04-roadmap/iteration-tracking.md` 中更新范围、验收和状态。
- 涉及端侧或服务端模块变更时，更新 `docs/03-modules/` 对应文档。
- 涉及 API、数据表、权限、设备、部署或测试策略时，更新 `docs/02-architecture/`、`docs/05-engineering/` 或 `docs/06-runbooks/` 对应文档。
- 如果没有文档变更，必须在提交或 PR 说明中明确写明“无文档影响”及原因。

## 代码质量要求

- 计划执行、策略评估、设备控制必须有自动化测试。
- 不允许只在 UI 上显示成功但服务端无真实状态。
- 不允许端侧复制复杂业务规则而不抽共享逻辑。
- 不允许吞掉外部上游错误而无日志。
