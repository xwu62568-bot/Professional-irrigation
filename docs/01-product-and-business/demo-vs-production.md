# 演示态与生产态边界

## 当前演示态能力

| 能力 | 当前状态 | 是否可直接生产使用 |
| --- | --- | --- |
| Web 管理界面 | 页面完整度较高，部分数据接 Supabase，部分仍有 mock/seed | 部分可用 |
| 小程序移动工作台 | 页面和 mini API 已具备，适合演示现场流程 | 部分可用 |
| Supabase Auth | Web 和 mini 登录链路已接入 | 可作为一期基础 |
| 地块/分区 | Web 可 CRUD，小程序可读取 | 需补审计和删除约束 |
| 设备 | 有真实表和 demo 设备混合逻辑 | 不可直接生产 |
| MQTT 控制 | 固定 Wi-Fi 演示设备可控制 | 不可直接生产 |
| 计划执行 | 有 run/step 和基础下发逻辑 | 需补测试、ACK 和失败处理 |
| 策略 | 读取展示和端侧表单存在 | 不可生产 |
| AI 助手 | Dify 代理链路可用 | 需补权限、审计和上下文 |
| MCP server | 工具骨架可用 | 需补安全和操作审计 |

## 不能混入正式域模型的内容

- `source=demo` 的固定设备。
- Web 端自动 seed 设备数据的逻辑。
- 小程序中“已确认（演示）”“已忽略（演示）”类提示。
- `mqtt-gateway-service` 对单一 `WIFI_DEMO_DEVICE_ID` 的限制。
- ET 占位算法和 mock 风险数据。
- 只返回 accepted 但不写数据库的策略动作。

## 生产化替换要求

设备链路：

- demo 设备入口保留在独立调试模块。
- 正式设备必须通过设备台账、设备凭证、通道绑定和回执事件管理。
- MQTT 回执必须落库，不能只保存在内存 state。

策略链路：

- 策略创建、启停、确认、忽略必须写库。
- 策略评估必须有输入、输出、触发原因和操作人。
- 自动执行必须生成可追踪 run 或 action。

权限链路：

- 当前单用户所有权模型需要升级为项目/组织/角色模型。
- Web 直连 Supabase 写业务的路径需要收敛到服务端 API 或明确保留边界。

测试链路：

- 计划执行状态机必须有自动化测试。
- 小程序 API 响应必须与 `packages/irrigation-api` 契约一致。
- MQTT 网关必须有失败、超时、非法设备、断连场景测试。
