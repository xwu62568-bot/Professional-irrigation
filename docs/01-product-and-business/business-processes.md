# 业务流程

## 完整目标闭环

```text
项目/用户
  -> 地块建档
  -> 分区划分
  -> 设备接入
  -> 分区设备绑定
  -> 计划或策略配置
  -> 执行触发
  -> 设备命令下发
  -> 回执和状态采集
  -> 告警/建议/复盘
  -> 下一轮计划优化
```

## 地块建档流程

1. 管理员在 Web 创建地块，录入名称、编号、作物、生育期、面积和边界。
2. 系统保存到 `fields`，并根据边界计算中心点和展示坐标。
3. 管理员为地块创建分区，保存到 `field_zones`。
4. 小程序读取 `/mini/fields` 和 `/mini/fields/:id` 展示只读地块和分区状态。

生产化要求：

- 地块边界修改需要保留审计记录。
- 删除地块前必须检查计划、策略、绑定和执行历史。
- 小程序不直接编辑复杂地块边界。

## 设备接入和绑定流程

1. 管理员登记控制器或传感器，写入 `irrigation_devices`。
2. 控制器按通道/站点绑定到地块分区，写入 `zone_device_bindings`。
3. 计划执行时根据分区查绑定，再按控制器 `client_key` 下发命令。
4. 传感器数据用于状态展示和后续策略评估。

当前限制：

- 固定 Wi-Fi 演示设备通过配置生成，并非完整生产设备入网流程。
- MQTT 回执主要保存在网关内存状态，尚未形成设备状态事件表。

生产化要求：

- 设备入网、解绑、替换、停用需要明确状态机。
- 控制器通道必须与分区绑定唯一或有明确优先级。
- 设备回执和在线状态需要持久化。

## 计划配置和执行流程

1. 用户创建计划，选择地块、周期、开始时间、模式、雨天策略和执行分区。
2. 系统写入 `irrigation_plans` 和 `irrigation_plan_zones`。
3. 用户手动启动，或调度器按时间自动触发（统一按项目时区计算）。
4. `run-service` 创建 `plan_runs` 和 `plan_run_steps`。
5. 每个 step 按顺序打开绑定阀门，等待时长，关闭阀门。
6. 用户可停止计划，服务端将 run 标记为 `cancel_requested` 并尽量关闭当前阀门。
7. 执行结果进入后续复盘和告警。

生产化要求：

- run 和 step 状态枚举必须固定并文档化。
- 命令下发、ACK、超时、失败重试需要可追溯。
- 已启动 run 不应被计划编辑隐式改变。
- `daily/weekly/interval` 的 `next_run_at` 必须按项目时区推导，不依赖“全表扫描取模”。
- 夏令时/DST 策略：若本地时间不存在（跳时）则顺延到下一个有效时刻；若本地时间重复（重时）只执行一次。

## 策略决策流程

目标流程：

1. 策略定义阈值或 ET 条件。
2. 策略评估器读取地块、分区、传感器、天气、计划和历史执行。
3. 生成策略建议或触发动作。
4. `suggest` 模式只记录建议。
5. `confirm` 模式等待人工确认。
6. `auto` 模式生成执行任务或临时计划。
7. 所有评估输入、输出和动作必须审计。

当前状态：

- 表、类型、列表、详情基本存在。
- 创建和动作接口未形成真实写库闭环。
- 小程序确认/忽略仍有演示痕迹。

## 告警和复盘流程

目标告警来源：

- 设备离线、低电、回执失败。
- 计划执行失败、超时、取消。
- 地块墒情异常、ET 缺水、降雨冲突。
- 服务端异常、上游 Dify 或 MQTT 不可用。

目标复盘内容：

- 本次计划执行了哪些分区。
- 每个分区实际开关阀时间。
- 下发了哪些命令，设备是否确认。
- 实际水量或估算水量。
- 异常原因和人工处理记录。
