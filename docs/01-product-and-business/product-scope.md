# 产品范围

## 产品定位

智慧灌溉平台面向农田、园区或灌溉项目的数字化管理，核心目标是把“地块建档、设备接入、分区绑定、灌溉计划、自动策略、执行回执、风险提醒、复盘分析”串成完整闭环。

当前代码已经具备演示和部分真实数据接入能力，但尚未达到完整生产系统标准。后续规划应明确区分“演示链路”“一期生产链路”和“长期平台能力”。

## 主要用户角色

| 角色 | 关注点 | 主要端 |
| --- | --- | --- |
| 农场/项目管理员 | 地块、设备、计划、策略、账号和项目配置 | Web |
| 调度人员 | 当天执行计划、异常确认、手动启停 | Web / 小程序 |
| 现场巡检人员 | 设备在线、阀门状态、地块状态、现场操作 | 小程序 |
| 运维/开发人员 | 服务状态、接口、设备链路、日志、部署 | 服务端 / 文档 |
| 产品/业务负责人 | 迭代范围、交付状态、生产化缺口 | 文档 / Web |

## 核心业务域

| 业务域 | 当前目标 | 生产化目标 |
| --- | --- | --- |
| 地块 | 建立地块、边界、作物、分区基础数据 | 支持多项目、多角色、边界版本和历史记录 |
| 分区 | 将地块拆分为可执行灌溉单元 | 与设备通道、策略、执行记录稳定关联 |
| 设备 | 管理控制器、传感器和演示设备 | 支持真实设备入网、状态同步、回执落库 |
| 计划 | 配置轮灌计划并手动/自动执行 | 支持冲突检测、执行审计、失败重试 |
| 策略 | 展示阈值/ET 策略，部分端侧表单已具备 | 形成评估、建议、确认、自动执行闭环 |
| AI 助手 | 代理 Dify 完成问答 | 结合项目上下文、运行数据和运维知识库 |
| 监控告警 | 当前以页面状态和日志为主 | 支持设备告警、计划异常、服务监控和通知 |

## 端侧职责

Web 管理端负责复杂配置和管理操作：地块地图编辑、分区维护、设备绑定、计划编排、策略配置、调试入口、数据分析。

小程序负责移动现场工作台：总览、地块/设备/计划查看、轻量启停、确认、巡检、AI 问答。小程序不应承载复杂地图绘制、大批量计划编排、系统权限治理。

服务端负责统一业务 API、认证会话、计划执行、设备网关聚合、AI 代理、MCP 能力和后续生产化审计。

## 范围边界

一期生产化应优先保证：

- 地块、分区、设备、计划的真实数据链路。
- 小程序统一走 Node API。
- 计划执行能生成可追溯 run、step、command 日志。
- demo 设备与正式设备隔离。
- 策略至少完成 CRUD 和启停，不再停留在端侧演示。

暂不作为一期重点：

- 复杂多租户计费。
- 大屏 BI。
- 完整工单系统。
- 多品牌设备通用接入平台。
- 高级 AI 自动决策闭环。
